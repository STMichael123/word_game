## 1. 研究动机

### 为什么做 StoryWeaver

- 课程学习目标匹配

  - 文本数据处理：采集、清洗、结构化、标注
  - 文本生成：上下文感知生成与可控生成
  - 交互式系统：意图理解、状态跟踪、实时响应
  - 专业产出：完整演示 + 规范技术写作

  

### 为什么选择交互式文字冒险场景

- 天然融合**自然语言理解（NLU）+ 自然语言生成（NLG）+ 记忆 / 状态管理**
- 易于评估：分支差异、一致性冲突、响应时间
- 比单纯文本生成更贴近真实 “生成式系统工程”

**重点不是写一个精彩故事，而是搭建一个健壮的、基于 NLP 的交互式叙事系统。**

------

## 2. 任务定义

### 要做什么

**StoryWeaver 任务定义**

- 基于 AI 的动态剧情生成文字冒险游戏
- 输入：玩家选择 / 玩家文本输入（如 “与长老谈判”“探索洞穴”）
- 系统：解析输入、维护叙事状态、生成连贯的下一段剧情
- 输出：下一段剧情（叙述 / 对话）+ 下一步选项（或自由输入引导）

**关键特性**

- 动态分支剧情
- 上下文连贯性
- 多轮交互

**主要挑战**

- 输入多样化
- 可控分支生成
- 长期一致性保持

**课堂要求**

- 8 分钟展示
- 现场演示
- 说明方法 + 效果

------

## 3. 核心 NLP 子任务（必须覆盖）

1. **自然语言理解（NLU）：意图识别**

   - 将玩家输入映射为动作 / 意图类型（谈判、探索、询问等）
   - 可选：槽位 / 实体抽取（角色、地点、物品、目标）
   - 低置信度处理：澄清、回退、安全选项

2. **剧情一致性维护**

   - 跟踪并保存事实（角色状态、物品、时间线、已揭示信息）
   - 核心能力：检测矛盾 + 修复 / 拒绝 + 解释

3. **自然语言生成（NLG）：上下文感知生成**

   - 生成必须反映之前事件、角色、地点、约束
   - 推荐结构化输出：叙述 + 下一步选项 + 对话
   - 分支可观测：不同选择 → 不同结果

   

------

## 4. 必须的开发步骤

1. **数据准备**
   - 收集整理：冒险脚本、分支叙事、对话数据、一致性标注
   - 预处理：清洗、剧情分段、标注叙事逻辑
   - 目标：可复现、可评估的训练 / 测试数据
2. **算法设计**
   - 上下文生成、意图识别、一致性检测、对话管理
   - 高效实时流程：延迟、缓存、上下文窗口控制
   - 分支策略：选择必须有效改变剧情
3. **系统实现**
   - 推荐技术栈：Transformers、PyTorch、Gradio
   - 交互流程：输入 → 生成 → 输出选项
   - 日志：记录轨迹用于评估与写作
4. **性能评估**
   - 叙事质量：连贯性 / 一致性
   - 响应性：响应时间
   - 选择匹配准确率 + 用户满意度

## 一句话版本

做一个**StoryWeaver**系统：

- 玩家每次输入一句话或选一个选项
- 系统要：
  1. 能看懂大概想干嘛（意图识别）
  2. 记住已经发生的故事（状态管理 / 一致性）
  3. 继续讲一段合理的剧情，并给出下一步选项（文本生成）
- 最后：做 8 分钟展示 + 现场跑一下系统 + 写 1 份 8 页左右的报告。

## 老师“最低要求”的功能（技术上）

### 1. 有一个**可玩的 Demo 系统**

技术栈老师已经帮你选好了：

- 后端：Python
- 模型：直接调用大模型 API（GPT / LLaMA 等），**不用训练自己的模型**
- 前端：Gradio 或 Streamlit（一个简单网页就够）

Demo 要做到：

1. 玩家可以输入：
   - 选项编号（例如：1 / 2 / 3）
   - 自然语言（例如：“与长老谈判”、“探索洞穴”、“去客栈打听消息”）
2. 系统每次回应时：
   - 输出一小段**连续的故事**（叙述/对话）
   - 再列出**2–4 个下一步行动选项**（1. xxx 2. xxx …）
3. 系统能**记住之前发生的事**，不能每次都当新故事开始。

只要做到这个程度，现场演示就不会挂。

------

### 2. 必须有的三个 NLP 模块（哪怕是简单版）

这三个是评分里的“硬指标”：

1. **NLU：意图识别**

   - 把玩家输入粗略分类，比如：
     - 探索：explore
     - 谈判/交流：negotiate
     - 战斗：combat
     - 打听消息：query
     - 拜师/加入阵营：join
   - 最简单做法：
     - 用 if-else 和关键词匹配（“探索”“explore”“逛一逛” → explore）
   - 想加分：再用一个小模型做分类，和规则比较效果。

2. **剧情一致性 / 状态管理**

   - 你要维护一个

      

     ```
     game_state
     ```

     （可以是 Python 字典），里面有：

     - 当前地点：`location`
     - 玩家基础属性：`health`、`reputation`…
     - 拥有的物品：`inventory`
     - 遇到的重要角色及关系（可选）
     - 最近发生的事件列表：`event_history`

   - 每一轮：

     - 把这份状态写进 prompt 给大模型，让它知道“故事已经进行到哪了”
     - 根据新生成的剧情，更新状态（比如扣血、换场景、声望+1）

3. **NLG：上下文感知生成**

   - 用大模型生成下一段故事：

     - 必须**参考 `game_state` 和 `event_history`**
     - 不能写出“之前死了的人又活了”等明显矛盾

   - 你在 system prompt 里强制：

     - 输出格式：

       text

       ```text
       [剧情叙述]
       ……（几段文字）
       
       [可选行动]
       1. …
       2. …
       3. …
       ```

     - 不同选项要有明显不同结果（不是换个表述而已）

------

## 老师希望看到的“开发过程”

不要求你真的大规模做数据集、训练 Transformer，但希望你**展示出有“工程思路”**。可以简化成：

1. **数据准备（可以很简单）**
   - 找几篇分支故事 / 冒险脚本 / 游戏对话当素材：
     - 用来分析：常见意图类型、场景切换、物品系统
   - 可以手工标注几条对话：
     - 哪一句属于哪种“意图”
     - 哪些地方出现“矛盾剧情”
2. **算法设计**
   - 画一张架构图：
     - 输入 → NLU（意图/实体）→ 状态管理（更新 game_state）→ NLG → 输出剧情+选项
   - 解释你们如何：
     - 控制上下文长度（只给模型最近 N 条事件）
     - 控制分支数量和差异（用 prompt 限制）
3. **系统实现**
   - 说明你们用的：
     - Python + 哪个大模型 API
     - Gradio / Streamlit
   - 解释关键函数：
     - `classify_intent`
     - `update_game_state`
     - `generate_story`
     - `game_step`（一次完整交互）
4. **性能评估（至少做一点）**
   - 叙事质量：
     - 找几条完整对话轨迹（多轮），人工打分：连贯性、矛盾程度
   - 响应时间：
     - 统计平均每轮响应时间（比如 1–3 秒）
   - 分支差异：
     - 定义一个简单指标：同一状态下，选不同选项，后续故事是否明显不同（人工+简单文本相似度）
   - 若有时间可以做个小问卷，让几个人体验后打分。
